# compiler 项目前置跨人协作接口约定

本文档用于完成第一步前置协作（短平快），只约定跨模块接口，不展开具体算法实现。  
适用范围：`compiler/src/main/java/edu/groupname/compiler/*` 当前已有目录与文件。

## 1. 文档目标与适用文件

本约定覆盖以下跨人协作链路：

1. A(词法) -> B(语法)
2. B(语法) -> C(语义)
3. C(语义) -> D(IR)
4. D(集成) -> 全员联调

对应已有文件（关键）：

- `lexer/Lexer.java`
- `lexer/LexerImpl.java`
- `lexer/LexicalAnalyzerResult.java`
- `common/token/Token.java`
- `common/token/TokenType.java`
- `parser/Parser.java`
- `parser/LR1Parser.java`
- `parser/ParserResult.java`
- `parser/ParseTraceStep.java`
- `semantic/SemanticAnalyzer.java`
- `semantic/SemanticAnalyzerImpl.java`
- `semantic/SemanticResult.java`
- `ir/IRGenerator.java`
- `ir/IRGeneratorImpl.java`
- `app/CompilerPipeline.java`

## 2. 总体阶段契约（强约束）

### 2.1 `CompilerPipeline` 调用顺序

以 `app/CompilerPipeline.java` 为唯一编排入口：

1. `lexer.analyze(sourceCode)`
2. `parser.parse(lexResult.tokens())`
3. `semanticAnalyzer.analyze(parserResult)`
4. `irGenerator.generate(semanticResult)`

### 2.2 错误短路策略（统一）

为避免无效级联错误，统一采用短路：

- 若词法有错：语法/语义/IR 不再进行真实处理，返回空结果容器。
- 若语法有错：语义/IR 不再进行真实处理，返回空结果容器。
- 若语义有错：IR 不再进行真实处理，返回空结果容器。

说明：短路策略由 D 在 `CompilerPipeline` 落地，A/B/C 保证各自结果结构可被安全读取。

## 3. A -> B 约定：Token 与终结符映射

### 3.1 `Token` 结构约定

沿用现有定义：

- `Token.type()`：来自 `TokenType`
- `Token.lexeme()`：源串字面值
- `Token.position()`：行列号，非法或未知可用 `SourcePosition.UNKNOWN`

### 3.2 `TokenType` 与文法终结符映射

语法分析阶段统一按下表解释词法输出：

- `TokenType.IDENTIFIER` -> `id`
- `TokenType.INTEGER_LITERAL` -> `num`
- `TokenType.REAL_LITERAL` -> `real`
- `TokenType.BOOLEAN_LITERAL` 且 lexeme=`true` -> `true`
- `TokenType.BOOLEAN_LITERAL` 且 lexeme=`false` -> `false`
- `TokenType.KEYWORD` -> 直接使用 `lexeme`（如 `if/else/while/do/break/int/float/bool`）
- `TokenType.OPERATOR` -> 直接使用 `lexeme`（如 `+ - * / = == != < <= > >= && || !`）
- `TokenType.DELIMITER` -> 直接使用 `lexeme`（如 `{ } ( ) [ ] ;`）
- `TokenType.EOF` -> `$`（解析器内部可用 `$` 表示输入结束）

### 3.3 词法输出约束

- `LexicalAnalyzerResult.tokens()` 必须保证末尾仅 1 个 EOF。
- 注释与空白不会形成 token。
- 非法字符或非法词素进入 `LexicalAnalyzerResult.errors()`（`LexicalError`）。

## 4. B -> C 约定：语法结果输入语义的最小信息

### 4.1 `ParserResult` 必须语义可消费

当前 `ParserResult` 已有字段：

- `accepted`
- `trace`
- `errors`

为适配语义分析，约定新增或扩展一项语法产物（由 B 实现，C 消费）：

- 方案首选：在 `ParserResult` 增加 `List<Integer> reducedProductionIndexes`
  - 含义：按规约顺序记录使用的产生式编号（编号与 `Grammar.productions()` 下标一致）。
- 备选方案：增加语法树根节点对象（若小组愿意新增 AST 类）。

本阶段建议采用首选方案，改动最小，便于与 LR 规约过程对齐。

### 4.2 `ParseTraceStep` 输出约定

`ParseTraceStep` 仅用于实验展示，不作为语义输入的唯一来源。  
语义输入必须来自 `ParserResult` 的结构化字段（如规约序列或 AST）。

### 4.3 语法错误约定

- `accepted=false` 时，`errors` 至少有 1 条 `SyntaxError`。
- 若 `accepted=false`，C 不做真实语义检查，直接返回空语义结果（见第 2 节短路策略）。

## 5. C -> D 约定：语义结果输入 IR 的最小信息

### 5.1 `SemanticResult` 扩展约定

当前 `SemanticResult` 仅有 `errors`，不足以支撑 IR。  
约定新增或扩展以下字段（由 C 实现，D 消费）：

- `List<Symbol> symbols`：全局可见符号快照（用于导出/调试）
- `List<SemanticAction>` 或等价结构：按语句顺序的中间语义动作
  - 最小动作集：声明、赋值、算术、条件跳转、循环、break
- 可选：`Map<String, String> expressionTypes`（表达式类型信息）

若不新增 `SemanticAction` 类，可先用 `List<String>` 临时承载，但需保证格式固定并写入本文档附录。

### 5.2 符号表约定

`symbol/ScopedSymbolTable.java` 语义如下：

- `define(Symbol)`：当前作用域定义
- `resolve(String)`：先当前作用域，再父作用域（若扩展多层作用域）

`Symbol.kind` 统一使用现有枚举：

- `VARIABLE`
- `ARRAY`
- `TEMPORARY`

### 5.3 语义错误约定

- 存在语义错误时，`SemanticResult.errors()` 非空。
- 有语义错误时，D 仅返回空 `IRProgram(List.of())`，不生成部分 IR。

## 6. D 侧集成落地约定

### 6.1 `IRGenerator` 输入前提

`IRGenerator.generate(SemanticResult)` 在调用前由 `CompilerPipeline` 保证：

- 若语义有错则不进入真实生成逻辑（或生成器内部二次防御）。

### 6.2 四元式最小输出规范

沿用现有 `ir/Opcode.java`：

- `ASSIGN`
- `ADD`
- `SUB`
- `MUL`
- `DIV`
- `IF_FALSE_GOTO`
- `GOTO`
- `LABEL`

`Quadruple(op, arg1, arg2, result)` 填充约定：

- 一元/赋值操作时未使用参数填空串 `""`。
- `LABEL`：`result` 放标签名，`arg1/arg2` 为空串。
- 跳转：目标标签放 `result`。

## 7. 命名与格式统一（避免返工）

### 7.1 布尔与关键字

- `true/false` 作为布尔字面量处理。
- `bool` 为类型关键字，不等同于 `BOOLEAN_LITERAL`。

### 7.2 数值字面量

- 整数 -> `INTEGER_LITERAL`（语法映射 `num`）
- 实数 -> `REAL_LITERAL`（语法映射 `real`）

### 7.3 输入结束符

- 词法层：`TokenType.EOF` + `<EOF>`
- 语法层：内部映射为 `$`
